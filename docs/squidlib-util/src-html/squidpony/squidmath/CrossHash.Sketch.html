010/**
011 * Simple hashing functions that we can rely on staying the same cross-platform.
012 * The static methods of this class (not its inner classes) use a custom algorithm
013 * designed for speed and general-purpose usability, but not cryptographic security;
014 * this algorithm is sometimes referred to as Wisp, and several other algorithms are
015 * available in static inner classes, some of which have different goals, such as
016 * reduced likelihood of successfully reversing a hash, or just providing another
017 * choice. The hashes this returns are always 0 when given null to hash. Arrays with
018 * identical elements of identical types will hash identically. Arrays with identical
019 * numerical values but different types will sometimes hash differently. This class
020 * always provides 64-bit hashes via hash64() and 32-bit hashes via hash(), and some
021 * of the algorithms here may provide a hash32() method that matches older behavior
022 * and uses only 32-bit math. The hash64() and hash() methods, at least in Wisp and
023 * Mist, use 64-bit math even when producing 32-bit hashes, for GWT reasons. GWT
024 * doesn't have the same behavior as desktop and Android applications when using ints
025 * because it treats doubles mostly like ints, sometimes, due to it using JavaScript.
026 * If we use mainly longs, though, GWT emulates the longs with a more complex
027 * technique behind-the-scenes, that behaves the same on the web as it does on
028 * desktop or on a phone. Since CrossHash is supposed to be stable cross-platform,
029 * this is the way we need to go, despite it being slightly slower.
030 * <br>
031 * There are several static inner classes in CrossHash: Lightning, Storm, Falcon, Mist, and Sketch,
032 * each providing different hashing properties, as well as the inner IHasher interface and a
033 * compatibility version of Wisp as a subclass. Older versions of SquidLib encouraged using a
034 * subclass because the non-nested-class methods used a lousy implementation of the FNV-1a algorithm,
035 * which was roughly 10x slower than the current methods in CrossHash and had poor correlation
036 * properties. In the current versio, you probably will be fine with the default functions in
037 * CrossHash, using the Wisp algorithm. If you need a salt to alter the hash function,
038 * using one of a large family of such functions instead of a single function like Wisp, then Mist
039 * is a good choice, and should probably be preferred over Storm since it allows a larger and more
040 * complex salt. Falcon and Lightning are mostly superseded by Wisp. Sketch is a work in progress,
041 * and is not encouraged for general use yet.
042 * <br>
043 * IHasher values are provided as static fields, and use Wisp to hash a specific type or fall
044 * back to Object.hashCode if given an object with the wrong type. IHasher values are optional
045 * parts of OrderedMap, OrderedSet, Arrangement, and the various classes that use Arrangement
046 * like K2 and K2V1, and allow arrays to be used as keys in those collections while keeping
047 * hashing by value instead of the normal hashing by reference for arrays. You probably won't
048 * ever need to make a class that implements IHasher yourself; for some cases you may want to
049 * look at the {@link Hashers} class for additional functions.
050 * <br>
051 * The inner classes provide alternate, faster hashing algorithms. Lightning, Wisp, and Falcon
052 * have no theoretical basis or grounds in any reason other than empirical testing for why they
053 * do what they do, and this seems to be in line with many widely-used hashes (see: The Art of
054 * Hashing, http://eternallyconfuzzled.com/tuts/algorithms/jsw_tut_hashing.aspx ). That said, Wisp
055 * performs very well, ahead of Arrays.hashCode (10.5 ms instead of 15 ms) for over a million
056 * hashes of 16-element long arrays, not including overhead for generating them, while SipHash and
057 * FNV-1a take approximately 80 ms and 135-155 ms, respectively, for the same data). Lightning and
058 * Falcon perform less-well, with Lightning taking 17 ms instead of 15 ms for Arrays.hashCode, and
059 * Falcon taking about 12.3 ms but slowing down somewhat if a 32-bit hash is requested from long
060 * data. All of these have good, low, collision rates on Strings and long arrays. Sketch is only
061 * slightly slower than Wisp, but offers little to no advantage over it yet.
062 * <br>
063 * Storm is a variant on Lightning with 64 bits for a salt-like modifier as a member variable,
064 * which can make 2 to the 64 individual hashing functions from one set of code. Mist is similar,
065 * but has 128 bits for some hashes (including any calls to hash64()) and 64 bits for some other
066 * hashes (only calls to hash() with data that doesn't involve long or double arrays). Storm and Mist
067 * have some properties of a cryptographic hash, but is not recommended it for that usage. It is,
068 * however ideal for situations that show up often in game development where end users may be able
069 * to see and possibly alter some information that you don't want changed (i.e. save data stored on
070 * a device or in the browser's LocalStorage). If you want a way to verify the data is what you
071 * think it is, you can store a hash, using one of the many-possible hash functions this can
072 * produce, somewhere else and verify that the saved data has the hash it did last time; if the
073 * exact hashing function isn't known (or exact functions aren't known) by a tampering user,
074 * then it is unlikely they can make the hash match even if they can edit it. Storm is slightly
075 * slower than Lightning just like Mist is slightly slower than Wisp, at about 20 ms for a
076 * million hashes with Storm instead of Lightning's 17 ms, and at about 18 ms for Mist for the same
077 * data instead of Wisp's 10.5, but should never be worse than twice as slow as Arrays.hashCode, and
078 * is still about three times faster than the similar SipHash that SquidLib previously had here.
079 * <br>
080 * All of the hashes used here have about the same rate of collisions on identical data
081 * (testing used Arrays.hashCode, all the hashes in here now, and the now-removed SipHash), with
082 * any fluctuation within some small margin of error. Wisp (typically via the non-nested methods
083 * in CrossHash) and Mist are the two most likely algorithms you might use here.
084 * <br>
085 * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
086 * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
087 * in Arrays.hashCode and the former default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing
088 * SipHash) did approximately as well as Lightning, with no clear patterns visible (Sip has been removed from
089 * SquidLib because it needs a lot of code and is slower than all of the current hashes). The idea is from
090 * <a href="http://www.clockandflame.com/media/Goulburn06.pdf">a technical report on visual uses for hashing (PDF)</a>.
091 * <ul>
092 * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
093 * <li>{@link CrossHash#hash(int[])}: http://i.imgur.com/x8SDqvL.png</li>
094 * <li>(Former) CrossHash.Sip.hash(int[]): http://i.imgur.com/keSpIwm.png</li>
095 * <li>{@link CrossHash.Lightning#hash(int[])}: http://i.imgur.com/afGJ9cA.png</li>
096 * </ul>
097 * <br>
098 * Note: This class was formerly called StableHash, but since that refers to a specific
099 * category of hashing algorithm that this is not, and since the goal is to be cross-
100 * platform, the name was changed to CrossHash.
101 * Note 2: FNV-1a was removed from SquidLib on July 25, 2017, and replaced as default with Wisp. This
102 * accompanied a general start toward preferring 64-bit math for GWT compatibility reasons, although
103 * that is counter-intuitive because 64-bit longs aren't native to JavaScript and GWT implements its
104 * own variety.
105 * <br>
106 * Created by Tommy Ettinger on 1/16/2016.
107 * @author Tommy Ettinger
108 */

556     * An interface that can be used to move the logic for the hashCode() and equals() methods from a class' methods to
557     * an implementation of IHasher that certain collections in SquidLib can use. Primarily useful when the key type is
558     * an array, which normally doesn't work as expected in Java hash-based collections, but can if the right collection
559     * and IHasher are used. See also {@link Hashers} for additional implementations, some of which need dependencies on
560     * things the rest of CrossHash doesn't, like a case-insensitive String hasher/equator that uses RegExodus to handle
561     * CharSequence comparison on GWT.
562     */
563    public interface IHasher extends Serializable {
564        /**
565         * If data is a type that this IHasher can specifically hash, this method should use that specific hash; in
566         * other situations, it should simply delegate to calling {@link Object#hashCode()} on data. The body of an
567         * implementation of this method can be very small; for an IHasher that is meant for byte arrays, the body could
568         * be: {@code return (data instanceof byte[]) ? CrossHash.Lightning.hash((byte[]) data) : data.hashCode();}
569         *
570         * @param data the Object to hash; this method should take any type but often has special behavior for one type
571         * @return a 32-bit int hash code of data
572         */
573        int hash(final Object data);
574
575        /**
576         * Not all types you might want to use an IHasher on meaningfully implement .equals(), such as array types; in
577         * these situations the areEqual method helps quickly check for equality by potentially having special logic for
578         * the type this is meant to check. The body of implementations for this method can be fairly small; for byte
579         * arrays, it looks like: {@code return left == right
580         * || ((left instanceof byte[] && right instanceof byte[])
581         * ? Arrays.equals((byte[]) left, (byte[]) right)
582         * : Objects.equals(left, right));} , but for multidimensional arrays you should use the
583         * {@link #equalityHelper(Object[], Object[], IHasher)} method with an IHasher for the inner arrays that are 1D
584         * or otherwise already-hash-able, as can be seen in the body of the implementation for 2D char arrays, where
585         * charHasher is an existing IHasher that handles 1D arrays:
586         * {@code return left == right
587         * || ((left instanceof char[][] && right instanceof char[][])
588         * ? equalityHelper((char[][]) left, (char[][]) right, charHasher)
589         * : Objects.equals(left, right));}
590         *
591         * @param left  allowed to be null; most implementations will have special behavior for one type
592         * @param right allowed to be null; most implementations will have special behavior for one type
593         * @return true if left is equal to right (preferably by value, but reference equality may sometimes be needed)
594         */
595        boolean areEqual(final Object left, final Object right);
596    }

598    /**
599     * Not a general-purpose method; meant to ease implementation of {@link IHasher#areEqual(Object, Object)}
600     * methods when the type being compared is a multi-dimensional array (which normally requires the heavyweight method
601     * {@link Arrays#deepEquals(Object[], Object[])} or doing more work yourself; this reduces the work needed to
602     * implement fixed-depth equality). As mentioned in the docs for {@link IHasher#areEqual(Object, Object)}, example
603     * code that hashes 2D char arrays can be done using an IHasher for 1D char arrays called charHasher:
604     * {@code return left == right
605     * || ((left instanceof char[][] && right instanceof char[][])
606     * ? equalityHelper((char[][]) left, (char[][]) right, charHasher)
607     * : Objects.equals(left, right));}
608     *
609     * @param left
610     * @param right
611     * @param inner
612     * @return
613     */
614    public static boolean equalityHelper(Object[] left, Object[] right, IHasher inner) {
615        if (left == right)
616            return true;
617        if ((left == null) ^ (right == null))
618            return false;
619        for (int i = 0; i < left.length && i < right.length; i++) {
620            if (!inner.areEqual(left[i], right[i]))
621                return false;
622        }
623        return true;
624    }

987    /**
988     * A quick, simple hashing function that seems to have good results. Like LightRNG, it stores a state that
989     * it updates independently of the output, and this starts at a large prime. At each step, it takes the
990     * current item in the array being hashed, adds a large non-prime used in LightRNG's generation function
991     * (it's 2 to the 64, times the golden ratio phi, and truncated to a signed long), multiplies by a prime
992     * called the "state multiplier", adds the result to the state and stores it, multiplies the value of the
993     * state by another prime called the "output multiplier", then XORs the current result with that value
994     * before moving onto the next item in the array. A finalization step XORs the result with a complex value
995     * made by adding the state (left over from the previous step) to what was the output multiplier, adding
996     * the last known value for result to the phi-related constant from LightRNG, multiplying that pair, adding
997     * the initial state (which turns out to be unusually good for this, despite no particularly special numeric
998     * qualities other than being a probable prime) and then bitwise-rotating it left by a seemingly-random
999     * number drawn from the highest 6 bits of the state.
1000     * <br>
1001     * This all can be done very quickly; a million hashes of a million different 16-element long arrays can be
1002     * computed in under 18-20 ms (in the benchmark, some amount of that is overhead from generating a new
1003     * array with LongPeriodRNG, since the benchmark uses that RNG's state for data, and the default
1004     * Arrays.hashCode implementation is only somewhat faster at under 16 ms). After several tries and tweaks
1005     * to the constants this uses, it also gets remarkably few hash collisions. On the same 0x100000, or
1006     * 1048576, RNG states for data, Lightning gets 110 collisions, the JDK Arrays.hashCode method gets 129
1007     * collisions, Sip (implementing SipHash) gets 145 collisions, and CrossHash (using the FNV-1a algorithm)
1008     * gets 135 collisions. Storm depends on the salt chosen, but with one initialized with the phi-based
1009     * constant that shows up in LightRNG and here, Storm gets 116 collisions. Dispersion is not perfect, but
1010     * at many bit sizes Lightning continues to have less collisions (it disperses better than the other hashes
1011     * with several quantities of bits, at least on this test data). Lightning also does relatively well, though
1012     * it isn't clearly ahead of the rest all the time, when hashing Strings, especially ones that use a larger
1013     * set of letters, it seems (FakeLanguageGen was used to make test data, and languages that used more
1014     * characters in their alphabet seemed to hash better with this than competing hashes for some reason).
1015     * <br>
1016     * There is certainly room for improvement with the specific numbers chosen; earlier versions used the
1017     * state multiplier "Neely's number", which is a probable prime made by taking the radix-29 number
1018     * "HARGNALLINSCLOPIOPEEPIO" (a reference to a very unusual TV show), truncating to 64 bits, and rotating
1019     * right by 42 bits. This version uses "Neely's number" for an initial state and during finalization, and
1020     * uses a different probable prime as the state multiplier, made with a similar process; it starts with the
1021     * radix-36 number "EDSGERWDIJKSTRA", then does the same process but rotates right by 37 bits to obtain a
1022     * different prime. This tweak seems to help with hash collisions. Extensive trial and error was used to
1023     * find the current output multiplier, which has no real relationship to anything else but has exactly 32 of
1024     * 64 bits set to 1, has 1 in the least and most significant bit indices (meaning it is negative and odd),
1025     * and other than that seems to have better results on most inputs for mystifying reasons. Earlier versions
1026     * applied a Gray code step to alter the output instead of a multiplier that heavily overflows to obfuscate
1027     * state, but that had a predictable pattern for most of the inputs tried, which seemed less-than-ideal for
1028     * a hash. Vitally, Lightning avoids predictable collisions that Arrays.hashCode has, like
1029     * {@code Arrays.hashCode(new long[]{0})==Arrays.hashCode(new long[]{-1})}.
1030     * <br>
1031     * The output multiplier is 0xC6BC279692B5CC83L, the state multiplier is 0xD0E89D2D311E289FL, the number
1032     * added to the state (from LightRNG and code derived from FastUtil, but obtained from the golden ratio
1033     * phi) is 0x9E3779B97F4A7C15L, and the starting state ("Neely's Number") is 0x632BE59BD9B4E019L.
1034     * <br>
1035     * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
1036     * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
1037     * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing
1038     * SipHash) does approximately as well as Lightning, with no clear patterns visible (Sip has been removed
1039     * from SquidLib because it needs a lot of code and is slower than Storm and especially Lightning). The
1040     * idea is from a technical report on visual uses for hashing,
1041     * http://www.clockandflame.com/media/Goulburn06.pdf .
1042     * <ul>
1043     * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
1044     * <li>{@link CrossHash#hash(int[])}: http://i.imgur.com/x8SDqvL.png</li>
