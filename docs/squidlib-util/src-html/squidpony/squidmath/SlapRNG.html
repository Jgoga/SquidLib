008/**
009 * Like PintRNG (uses int math primarily, making it good with GWT and 32-bit machines) and FlapRNG, but with a longer
010 * period than either, plus this implements StatefulRandomness. SlapRNG should have higher quality than FlapRNG based on
011 * some rough metrics, but its period is known to be longer than FlapRNG's for most seeds, if not all. Precisely how
012 * much longer isn't clear yet; this uses an irregular increment that depends on part of the current state, and it seems
013 * possible that certain rare seeds have an effect on the length of the period. That said, the period should be at least
014 * 2 to the 35 for most seeds, and is probably significantly higher. This generator is not nearly as fast as FlapRNG or
015 * PintRNG, but it has a longer (at least 8x) period, mainly thanks to more bits of state, and good quality even with
016 * similar seeds. The mechanism by which this produces pseudo-randomness is somewhat unusual, and involves two
017 * state variables as ints, one of which is used both before and after that state part changes. This slows down the
018 * generation slightly due to a data dependency between the before and after states (the same speed problem most linear
019 * congruential generators face when trying to reach ideal performance), but seems to significantly improve period. It
020 * should be noted that when targeting GWT and 32-bit machines, SlapRNG provides one of the better combinations of
021 * period, quality, compatibility, and speed that we have, emphasizing period (which is hard to do using 32-bit math)
022 * and quality. FlapRNG is faster by about 2x-2.5x, but sacrifices quality and period. PintRNG has better quality, is
023 * a little faster, and has a lower period. Other generators give up efficient compatibility with GWT due to long math
024 * (which is slower and probably produces garbage on GWT), though many are faster on PCs (LapRNG and ThunderRNG, for
025 * instance, completely beat SlapRNG on speed and period, and LapRNG has comparable quality).
026 * <br>
027 * Created by Tommy Ettinger on 5/29/2017.
028 */
029@Beta
030public class SlapRNG implements StatefulRandomness, Serializable {
031    private static final long serialVersionUID = 1L;
032    public SlapRNG(){
033        this((int)((Math.random() * 2.0 - 1.0) * 0x80000000),
034                (int)((Math.random() * 2.0 - 1.0) * 0x80000000));
035    }
036    public SlapRNG(final int seed) {
037        state0 = seed;
038        state1 = ((seed ^ 0xD0E89D2D) >>> 19 | (seed ^ 0xD0E89D2D) << 13);
039        state1 ^= state1 >>> (4 + (state1 >>> 28));
040        state1 *= 277803737;
041        state1 ^= (state1 >>> 22);
042    }
043    public SlapRNG(final int seed0, final int seed1) {
044        state0 = seed0;
045        state1 = seed1;
046    }
047    public SlapRNG(final long seed) {
048        state0 = (int)(seed & 0xFFFFFFFFL);
049        state1 = (int)(seed >>> 32);
050    }
051    public SlapRNG(final CharSequence seed)
052    {
053        this(CrossHash.Wisp.hash64(seed));
054    }

064        return (long)(state1) << 32 | (state0 & 0xFFFFFFFFL);

075        state1 = (int)(state >>> 32);

132    /**
133     * Using this method, any algorithm that needs to efficiently generate more
134     * than 32 bits of random data can interface with this randomness source.
135     * This implementation produces a different result than calling {@link #nextInt()} twice and shifting the bits
136     * to make a long from the two ints, which is what most int-centric generators do. The technique this uses, as this
137     * class usually does, reduces quality but sacrifices as little speed as possible. You get a long from this with
138     * only slightly longer time than it takes to produce than an int, from a primarily-int generator! Hooray. The
139     * downside is that only 2 to the 32 longs can be produced by this method (not the full 2 to the 64 range that would
140     * be ideal), though the period is a little higher than that (2 to the 33). It may be important to note that this
141     * changes the sequence of random numbers exactly in the same way as calling {@link #nextInt()}, so you could
142     * employ any combination of nextInt() and nextLong() calls and get the same result on a subsequent nextInt() call,
143     * given the same starting state.
144     * <p>
145     * Pseudo-random results may be between between Long.MIN_VALUE and Long.MAX_VALUE (both inclusive).
146     *
147     * @return a random long between Long.MIN_VALUE and Long.MAX_VALUE (both inclusive)
148     */

192     * @return a copy of this RandomnessSource

194    @Override

196        return new SlapRNG(state0, state1);

200    /**
201     * A simple "output stage" applied to state; this method does not update state on its own. If you expect to call
202     * this method more than once, you should perform some change to state as part of the call; a simple way to do this
203     * is to call this method like {@code FlapRNG.determine(state += 0x9E3779B9)}. The int 0x9E3779B9 is derived from
204     * the golden ratio, and shows up often as an optimal part of hashes and random number generators, but the constant
205     * can be any odd-number int, preferably a large one. This method doesn't offer very good quality assurances, but
206     * should be very fast.
207     * @param state should be changed when you call this (see above), e.g. {@code state += 0x9E3779B9}
208     * @return an altered version of state that should be very fast to compute but doesn't promise great quality
209     */
210    public static int determine(final int state)
211    {
212        return (state + (state << 8 ^ (state + 0xCABC2C8D) >>> (state >>> 29)));
213    }
214    /**
215     * A simple "output stage" applied to a two-part state like what FlapRNG uses normally; this method does not update
216     * state0 or state1 on its own. If you expect to call this method more than once, you should perform some change to
217     * state as part of the call; a simple way to do this is to call this method like
218     * {@code (state0 += FlapRNG.determine(state0, state1 += 0x9E3779B9))}. The int 0x9E3779B9 is derived from
219     * the golden ratio, and shows up often as an optimal part of hashes and random number generators, but the constant
220     * can be any odd-number int, preferably a large one. This method doesn't offer very good quality assurances, but
221     * should be very fast.
222     * @param state0 should be changed when you call this (see above), e.g. by adding the result to state0
223     * @param state1 should be changed when you call this (see above), e.g. {@code state1 += 0x9E3779B9}
224     * @return an altered version of state0/state1 that should be very fast to compute but doesn't promise great quality
225     */
226    public static int determine(final int state0, final int state1)
227    {
228        return (state1 + (state1 << 8 ^ (state0 + 0xCABC2C8D) >>> (state1 >>> 29)));
229    }
